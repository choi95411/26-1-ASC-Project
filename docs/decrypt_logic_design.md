# 복호화 로직 설계

## 1. 문서 목적

본 문서는 Rhysida 랜섬웨어의 암호화 구조 분석을 바탕으로 복호화 로직의 전체 흐름을 설계하는 것을 목적으로 한다.

본 프로젝트의 복호화 로직은 실제 피해 파일 복구를 보장하는 완성형 복구도구가 아니다. 이 문서는 Rhysida의 암호화 구조를 이해하고, 공개 연구 및 복구 원리에서 설명된 조건을 바탕으로 안전한 테스트 파일 환경에서 복호화 파이프라인을 재현하기 위한 분석용 설계 문서이다.

특히 본 문서에서는 ChaCha20을 파일 본문 복호화 알고리즘으로 다루지 않는다. 본 프로젝트에서 정리하는 구조에서는 ChaCha20이 PRNG로 사용되어 AES-CTR key와 IV를 생성하고, 실제 파일 본문 복호화는 AES-CTR로 수행된다.

## 2. 복호화 로직의 기본 방향

Rhysida 복호화는 암호화에 사용된 AES key와 IV를 확보하거나 재구성할 수 있는 경우에만 가능하다.

따라서 복호화 로직은 다음 방향으로 설계한다.

1. 암호화된 파일을 입력받는다.
2. 파일 구조를 분석하여 본문 영역과 footer 또는 metadata 영역을 분리한다.
3. 파일 끝에 append된 RSA-encrypted key/IV 영역을 확인한다.
4. 테스트 환경에서는 복구 가능한 key/IV 또는 재현된 key/IV 후보를 사용한다.
5. AES-CTR 방식으로 파일 본문 복호화를 시도한다.
6. 복호화 결과가 정상 파일인지 검증한다.
7. 성공한 결과는 별도 출력 경로에 저장한다.
8. 실패한 경우 원본 파일을 보존하고 로그를 남긴다.

전체 방향은 다음과 같이 정리할 수 있다.

```text
암호화 파일 입력
→ 파일 구조 분석
→ 암호화 본문과 footer 영역 분리
→ AES key/IV 확보 또는 후보 생성
→ AES-CTR 복호화
→ 결과 검증
→ 성공 파일 저장 또는 실패 로그 기록
```

## 3. 입력값

복호화 로직에서 사용할 수 있는 입력값은 다음과 같다.

| 입력값          | 설명                                       |
| ------------ | ---------------------------------------- |
| 암호화된 파일      | `.rhysida` 확장자를 가진 테스트 암호화 파일            |
| 입력 디렉토리      | 복호화 대상 파일이 저장된 경로                        |
| 출력 디렉토리      | 복호화 결과를 저장할 별도 경로                        |
| 파일 메타데이터     | 파일 크기, 생성 시간, 수정 시간 등                    |
| footer 후보 영역 | 파일 끝에 append된 RSA-encrypted key/IV 후보 영역 |
| key/IV 후보    | 테스트 환경에서 확보하거나 PRNG 재현으로 생성한 AES key/IV  |
| 파일 형식 정보     | 정상 파일 여부를 확인하기 위한 파일 시그니처 정보             |

본 프로젝트에서는 실제 피해 파일이 아니라 안전한 테스트 파일을 대상으로 한다.

## 4. 출력값

복호화 로직의 출력값은 다음과 같다.

| 출력값       | 설명                                  |
| --------- | ----------------------------------- |
| 복호화 결과 파일 | AES-CTR 복호화 결과로 생성된 파일              |
| 검증 결과     | 파일 시그니처, 해시, 크기 등을 기준으로 한 성공/실패 여부  |
| 로그 파일     | 처리 파일명, key/IV 후보, 성공 여부, 오류 메시지 기록 |
| 분석 결과     | 복호화 가능 조건과 실패 원인 정리                 |

복호화 결과는 원본 암호화 파일을 덮어쓰지 않고 별도 출력 디렉토리에 저장한다.

## 5. 전체 처리 흐름

복호화 로직의 전체 흐름은 다음과 같다.

```text
[입력 디렉토리]
        |
        v
[암호화 파일 목록 수집]
        |
        v
[파일 메타데이터 확인]
        |
        v
[파일 구조 분석]
        |
        v
[암호화 본문 / footer 영역 분리]
        |
        v
[AES key/IV 확보 또는 후보 생성]
        |
        v
[AES-CTR 복호화 시도]
        |
        v
[복호화 결과 검증]
        |
        v
[성공 파일 저장 / 실패 로그 기록]
```

## 6. 단계별 설계

### 6.1 파일 목록 수집

입력 디렉토리에서 복호화 대상이 되는 파일 목록을 수집한다.

확인할 항목은 다음과 같다.

1. 파일명
2. 파일 확장자
3. 파일 크기
4. 파일 경로
5. 생성 시간
6. 수정 시간
7. `.rhysida` 확장자 여부

이 단계에서는 파일을 수정하지 않고 읽기 전용으로 정보만 수집한다.

### 6.2 메타데이터 확인

파일 메타데이터는 암호화 시점 추정과 PRNG 시드 후보 범위 설정에 참고할 수 있다.

확인할 수 있는 메타데이터는 다음과 같다.

1. 파일 생성 시간
2. 파일 수정 시간
3. 암호화된 파일의 크기
4. 암호화 전후 확장자 변화
5. 동일 시점에 암호화된 파일 묶음
6. 파일 처리 순서 추정 가능 여부

단, 메타데이터만으로 복호화 성공을 보장할 수는 없다. 메타데이터는 key/IV 후보 범위를 좁히기 위한 보조 정보로 사용한다.

### 6.3 파일 구조 분석

Rhysida로 암호화된 파일은 개념적으로 다음 구조를 가진다.

```text
+-----------------------------+
| AES-CTR Encrypted File Data |
+-----------------------------+
| RSA-4096 Encrypted Key/IV   |
+-----------------------------+
```

파일 구조 분석 단계에서는 파일 본문 영역과 파일 끝에 append된 footer 또는 metadata 영역을 분리한다.

확인할 항목은 다음과 같다.

1. 전체 파일 크기
2. 암호화된 본문 영역
3. footer 후보 영역
4. RSA-encrypted key/IV 후보 영역
5. 파일 끝부분의 고정 크기 또는 패턴 존재 여부
6. 원본 파일 형식 추정 가능 여부

### 6.4 AES key/IV 확보 또는 후보 생성

Rhysida의 실제 파일 본문 암호화는 AES-CTR로 수행되므로, 복호화를 위해서는 AES key와 IV가 필요하다.

AES key와 IV는 다음 방식으로 확보하거나 후보를 생성할 수 있다.

1. 테스트 환경에서 생성한 key/IV를 기록해 사용하는 방식
2. ChaCha20 PRNG 출력을 재현하여 key/IV 후보를 생성하는 방식
3. 암호화 시점과 시드 후보를 바탕으로 가능한 key/IV 후보를 탐색하는 방식
4. 파일 구조와 알려진 파일 시그니처를 이용해 후보를 검증하는 방식

본 프로젝트의 demo에서는 실제 피해 파일 복구가 아니라 안전한 테스트 파일 환경을 사용하므로, 복구 가능한 key/IV 또는 재현 가능한 key/IV 후보를 기반으로 복호화 흐름을 검증한다.

### 6.5 AES-CTR 복호화 시도

생성되거나 확보된 AES key와 IV를 이용해 AES-CTR 복호화를 시도한다.

중요한 점은 다음과 같다.

```text
ChaCha20 복호화가 아니라 AES-CTR 복호화를 수행한다.
ChaCha20은 key/IV 생성을 위한 PRNG 역할로만 사용된다.
```

AES-CTR 복호화 단계에서는 원본 암호화 파일을 직접 수정하지 않고, 임시 출력 파일 또는 별도 출력 디렉토리에 결과를 저장한다.

처리 방식은 다음과 같다.

1. 암호화된 파일 본문 영역을 읽는다.
2. AES key와 IV를 설정한다.
3. AES-CTR 복호화를 수행한다.
4. 복호화 결과를 임시 파일로 저장한다.
5. 검증 단계로 전달한다.

### 6.6 복호화 결과 검증

복호화 결과가 정상 파일인지 검증한다.

검증 기준은 다음과 같다.

1. 파일 시그니처 확인
2. 확장자와 파일 헤더 일치 여부 확인
3. 파일 크기 확인
4. 원본 테스트 파일과 해시 비교
5. 일부 알려진 구조와 비교
6. 텍스트 파일의 경우 읽을 수 있는 문자열 비율 확인

테스트 파일 환경에서는 원본 파일을 알고 있으므로 해시 비교를 통해 복호화 성공 여부를 명확하게 확인할 수 있다.

예시는 다음과 같다.

```text
원본 파일 SHA-256 == 복호화 결과 SHA-256
→ 복호화 성공

원본 파일 SHA-256 != 복호화 결과 SHA-256
→ 복호화 실패 또는 key/IV 불일치
```

### 6.7 성공 파일 저장 및 실패 로그 기록

복호화가 성공하면 결과 파일을 별도 출력 경로에 저장한다.

복호화가 실패하면 다음 정보를 로그로 남긴다.

1. 파일명
2. 파일 경로
3. 파일 크기
4. 사용한 key/IV 후보 정보
5. 실패 원인
6. 예외 메시지
7. 처리 시간

실패한 경우에도 원본 암호화 파일은 절대 수정하지 않는다.

## 7. 안전성 고려사항

복호화 로직 설계에서 가장 중요한 점은 안전성과 원본 보존이다.

다음 원칙을 적용한다.

1. 원본 암호화 파일은 절대 덮어쓰지 않는다.
2. 복호화 결과는 별도 출력 디렉토리에 저장한다.
3. 실패한 복호화 결과는 구분하여 저장하거나 삭제한다.
4. 모든 작업 과정은 로그로 남긴다.
5. 일부 파일 복호화 실패가 전체 작업 중단으로 이어지지 않도록 한다.
6. 실제 피해 파일 복구를 보장하지 않는다.
7. KISA 공식 복구도구를 복제하지 않는다.
8. 테스트 파일 기반 demo로만 구현한다.

## 8. 코드 모듈 설계

복호화 demo는 다음 모듈로 나눌 수 있다.

```text
src/
├── rhysida_file_parser.py
├── safe_test_encrypt.py
├── recovery_demo.py
└── README.md
```

각 파일의 역할은 다음과 같다.

| 파일                       | 역할                                          |
| ------------------------ | ------------------------------------------- |
| `rhysida_file_parser.py` | 암호화된 테스트 파일의 본문 영역과 footer 영역을 분리           |
| `safe_test_encrypt.py`   | 안전한 테스트 파일을 AES-CTR 방식으로 암호화하고 key/IV를 기록   |
| `recovery_demo.py`       | 확보된 key/IV 또는 후보 key/IV를 이용해 AES-CTR 복호화 수행 |
| `src/README.md`          | demo 실행 순서와 입력/출력 설명                        |

## 9. 복호화 demo 예상 흐름

안전한 테스트 파일 기반 demo의 예상 흐름은 다음과 같다.

```text
1. 원본 테스트 파일 준비
2. safe_test_encrypt.py 실행
3. AES key/IV 생성
4. 테스트 파일을 AES-CTR로 암호화
5. key/IV를 테스트용 metadata로 저장
6. rhysida_file_parser.py로 파일 구조 확인
7. recovery_demo.py로 AES-CTR 복호화 수행
8. 원본 파일과 복호화 결과 해시 비교
9. 성공/실패 로그 출력
```

이 demo는 Rhysida의 실제 감염 동작을 구현하는 것이 아니라, 공개 원리 기반으로 암호화/복호화 파이프라인을 안전하게 재현하기 위한 것이다.

## 10. 한계

본 복호화 로직은 다음 한계를 가진다.

1. 실제 피해 파일 복구를 보장하지 않는다.
2. RSA 개인키 없이 실제 key/IV 복원이 불가능할 수 있다.
3. PRNG 시드 또는 내부 상태를 추정할 수 없으면 key/IV 재구성이 어렵다.
4. 파일 메타데이터가 손상되었거나 부족하면 후보 범위를 좁히기 어렵다.
5. 실제 Rhysida 샘플의 변종에 따라 파일 구조가 다를 수 있다.
6. 본 프로젝트는 KISA 공식 복구도구를 복제하지 않는다.

## 11. 정리

본 복호화 로직은 Rhysida 암호화 구조를 이해하고, 안전한 테스트 파일 환경에서 복호화 파이프라인을 재현하기 위한 설계이다.

핵심 구조는 다음과 같다.

```text
ChaCha20 PRNG → AES key/IV 생성
AES-CTR → 파일 본문 암호화 및 복호화
RSA-4096 → AES key/IV 보호
```

따라서 복호화 로직의 핵심은 ChaCha20 복호화가 아니라 AES-CTR 복호화이다. ChaCha20은 AES-CTR에 필요한 key와 IV를 생성하는 PRNG 역할로 분석한다.

본 프로젝트는 실제 피해 파일 복구를 보장하지 않으며, 공개 원리 기반의 교육용/분석용 복호화 재현 demo를 목표로 한다.
