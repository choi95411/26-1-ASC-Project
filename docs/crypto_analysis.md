# Rhysida 암호화 알고리즘 분석

## 1. 분석 목적

본 문서는 Rhysida 랜섬웨어의 파일 암호화 구조를 이해하고, 복호화가 가능해지는 조건을 정리하는 것을 목적으로 한다.

Rhysida 복호화 분석의 핵심은 암호 알고리즘 자체를 깨는 것이 아니다. 분석의 핵심은 파일 암호화에 사용된 AES key와 IV가 어떤 방식으로 생성되었는지, 그리고 해당 key/IV를 어떤 조건에서 재구성하거나 복원할 수 있는지를 파악하는 것이다.

본 프로젝트에서는 Rhysida의 암호화 구조를 다음 관점에서 분석한다.

1. PRNG 기반 난수 생성 구조
2. ChaCha20의 역할
3. AES-CTR 파일 암호화 방식
4. RSA-4096을 통한 key/IV 보호 구조
5. 복호화 가능 조건과 한계

## 2. Rhysida 암호화 흐름 개요

본 프로젝트에서 정리하는 Rhysida의 암호화 흐름은 다음과 같다.

```text
1. 암호화 대상 파일 탐색
2. 암호화 대상 파일 선별
3. srand(time(NULL)) 기반 시드 설정
4. LibTomCrypt ChaCha20 PRNG 초기화
5. 80바이트 난수 생성
6. 앞 32바이트를 AES-CTR key로 사용
7. 다음 16바이트를 AES-CTR IV로 사용
8. 파일 본문을 AES-CTR로 암호화
9. AES key와 IV를 RSA-4096으로 암호화
10. RSA로 암호화된 key/IV 데이터를 파일 끝에 append
11. 파일 확장자를 .rhysida로 변경
12. 랜섬노트 또는 관련 흔적 생성
```

이 중 복호화 가능성과 직접적으로 관련되는 부분은 다음 세 가지이다.

```text
ChaCha20 PRNG → AES key/IV 생성
AES-CTR → 실제 파일 본문 암호화
RSA-4096 → AES key/IV 보호
```

따라서 Rhysida 암호화 분석에서는 ChaCha20, AES-CTR, RSA-4096의 역할을 구분해서 이해해야 한다.

## 3. PRNG의 역할

PRNG는 Pseudo Random Number Generator의 약자로, 난수처럼 보이는 값을 생성하는 알고리즘이다. Rhysida 분석에서 PRNG가 중요한 이유는 파일 암호화에 사용되는 AES key와 IV 생성 과정에 관여하기 때문이다.

Rhysida의 암호화 구조에서는 `srand(time(NULL))`를 통해 시간 기반 시드를 설정하고, 이후 ChaCha20 기반 PRNG를 이용해 80바이트의 난수 데이터를 생성하는 구조로 정리할 수 있다.

이 80바이트 난수 데이터는 다음과 같이 사용된다.

| 구간       |      크기 | 역할             |
| -------- | ------: | -------------- |
| 0~31바이트  |   32바이트 | AES-CTR key    |
| 32~47바이트 |   16바이트 | AES-CTR IV     |
| 이후 영역    | 나머지 바이트 | 기타 내부 사용 가능 영역 |

즉, PRNG 분석의 목적은 AES-CTR에 사용되는 key와 IV가 어떻게 생성되었는지 파악하는 것이다. 만약 암호화 시점, 시드 후보, PRNG 내부 상태, 파일 처리 순서 등을 추정할 수 있다면 key/IV 후보를 재구성할 가능성이 생긴다.

## 4. ChaCha20의 역할

ChaCha20은 일반적으로 스트림 암호로 사용될 수 있는 알고리즘이다. 그러나 본 프로젝트에서 정리하는 Rhysida 암호화 구조에서는 ChaCha20이 파일 본문을 직접 암호화하는 용도가 아니라 PRNG 역할로 사용된다.

따라서 다음과 같이 구분해야 한다.

```text
잘못된 해석:
ChaCha20이 파일 본문을 직접 암호화한다.

수정된 해석:
ChaCha20은 PRNG로 사용되어 AES-CTR에 필요한 key와 IV를 생성한다.
실제 파일 본문 암호화는 AES-CTR로 수행된다.
```

ChaCha20 PRNG의 역할은 다음과 같다.

1. 시간 기반 시드 또는 내부 상태를 바탕으로 난수 데이터를 생성한다.
2. 생성된 난수 중 일부가 AES key와 IV로 사용된다.
3. 파일 본문 암호화 자체는 AES-CTR이 담당한다.
4. 복호화 가능성은 ChaCha20 PRNG 출력값을 재구성할 수 있는지와 연결된다.

따라서 본 프로젝트에서 ChaCha20 분석의 핵심은 ChaCha20 keystream으로 파일을 직접 복호화하는 것이 아니라, ChaCha20 PRNG가 생성한 AES key와 IV를 재현할 수 있는지 확인하는 것이다.

## 5. AES-CTR의 역할

AES-CTR은 Rhysida에서 실제 파일 본문을 암호화하는 데 사용되는 대칭키 암호 방식이다.

CTR 모드는 Counter Mode의 약자로, 블록 암호인 AES를 스트림 암호처럼 사용할 수 있게 하는 운용 모드이다. AES-CTR에서는 key와 IV 또는 counter 값을 바탕으로 암호화 스트림을 생성하고, 이를 평문과 결합하여 암호문을 만든다.

Rhysida 암호화 구조에서 AES-CTR의 입력값은 다음과 같다.

| 항목         | 설명                          |
| ---------- | --------------------------- |
| AES key    | ChaCha20 PRNG 출력 중 앞 32바이트  |
| IV         | ChaCha20 PRNG 출력 중 다음 16바이트 |
| Plaintext  | 암호화 대상 파일 본문                |
| Ciphertext | AES-CTR로 암호화된 파일 본문         |

AES-CTR의 특징상 암호화와 복호화는 같은 key/IV를 사용한다. 따라서 올바른 AES key와 IV를 확보하거나 재구성할 수 있다면 암호화된 파일 본문을 복호화할 수 있다.

반대로 AES key와 IV를 확보할 수 없다면 AES-CTR 자체를 직접 공격해서 복구하는 것은 현실적으로 어렵다.

## 6. RSA-4096의 역할

Rhysida는 파일 본문을 AES-CTR로 암호화한 뒤, AES key와 IV를 RSA-4096으로 암호화하여 파일 끝에 append하는 구조로 정리할 수 있다.

이때 RSA-4096의 역할은 파일 본문 전체를 암호화하는 것이 아니라, 파일 본문 암호화에 사용된 AES key와 IV를 보호하는 것이다.

전체 구조는 다음과 같이 볼 수 있다.

```text
[암호화된 파일 본문]
+
[RSA-4096으로 암호화된 AES key/IV]
```

이 구조에서 공격자는 RSA 공개키로 AES key/IV를 암호화해 파일 끝에 저장한다. 일반적인 상황에서는 RSA 개인키가 없으면 해당 key/IV를 직접 복원하기 어렵다.

따라서 복호화 가능성은 다음 조건과 연결된다.

1. RSA 개인키를 확보할 수 있는가
2. RSA로 보호되기 전의 AES key/IV를 다른 방식으로 재구성할 수 있는가
3. PRNG 시드나 내부 상태를 추정할 수 있는가
4. 암호화 시점과 파일 메타데이터를 활용할 수 있는가

본 프로젝트는 RSA 개인키를 확보하는 공격을 목표로 하지 않는다. 대신 공개 연구와 복구 원리에서 설명된 조건을 바탕으로 안전한 테스트 파일 환경에서 key/IV 재현 가능성과 복호화 파이프라인을 실험적으로 정리한다.

## 7. 파일 구조 관점

Rhysida로 암호화된 파일은 다음과 같은 구조로 이해할 수 있다.

```text
+-----------------------------+
| AES-CTR Encrypted File Data |
+-----------------------------+
| RSA-4096 Encrypted Key/IV   |
+-----------------------------+
```

분석 도구는 파일 끝부분에 추가된 RSA-encrypted metadata 또는 footer 영역을 확인하고, 파일 본문과 key/IV 보호 영역을 분리할 수 있어야 한다.

본 프로젝트의 파일 구조 parser는 다음 정보를 확인하는 것을 목표로 한다.

1. 암호화된 파일 전체 크기
2. 파일 본문 영역
3. footer 또는 metadata 후보 영역
4. RSA-encrypted key/IV 후보 영역
5. `.rhysida` 확장자 여부
6. 파일 시그니처 및 원본 형식 추정 가능 여부

## 8. 복호화 가능 조건

Rhysida로 암호화된 파일을 복호화하려면 다음 조건 중 일부가 충족되어야 한다.

1. AES key와 IV를 확보할 수 있어야 한다.
2. RSA로 암호화되기 전의 key/IV를 재구성할 수 있어야 한다.
3. PRNG 시드 또는 내부 상태를 추정할 수 있어야 한다.
4. 암호화 시점에 대한 시간 정보가 남아 있어야 한다.
5. 파일 처리 순서나 암호화 로그가 남아 있어야 한다.
6. 파일 포맷의 시그니처를 통해 복호화 성공 여부를 검증할 수 있어야 한다.
7. 안전한 테스트 환경에서 동일한 방식의 암호화/복호화 흐름을 재현할 수 있어야 한다.

이 조건이 충족되지 않으면 일반적인 방식으로 복호화는 어렵다.

## 9. 분석 방향

본 프로젝트에서는 다음 항목을 중심으로 암호화 알고리즘을 분석한다.

| 분석 항목          | 설명                                      |
| -------------- | --------------------------------------- |
| PRNG 구조        | 난수 생성 방식과 내부 상태 복원 가능성 확인               |
| ChaCha20 역할    | 파일 본문 암호화가 아니라 PRNG로 사용되는 구조 정리         |
| AES key/IV 생성  | PRNG 출력에서 AES key와 IV가 추출되는 방식 정리       |
| AES-CTR 암호화    | 실제 파일 본문 암호화 방식 분석                      |
| RSA-4096 보호 구조 | AES key/IV가 RSA로 암호화되어 파일 끝에 저장되는 구조 분석 |
| 파일 구조          | 암호화된 본문과 footer 영역 분리 가능성 확인            |
| 복호화 조건         | 어떤 조건에서 복호화가 가능한지 정리                    |
| 한계             | 실제 피해 파일 복구 보장 불가 및 KISA 도구 복제 아님을 명시   |

## 10. 정리

Rhysida 복호화 분석의 핵심은 ChaCha20으로 파일 본문을 직접 복호화하는 것이 아니다.

본 프로젝트에서 정리하는 구조는 다음과 같다.

```text
ChaCha20 PRNG → AES key/IV 생성
AES-CTR → 파일 본문 암호화
RSA-4096 → AES key/IV 보호
```

따라서 복호화 가능성은 AES-CTR에 사용된 key와 IV를 확보하거나 재구성할 수 있는지에 달려 있다. 본 프로젝트는 실제 피해 파일 복구를 보장하지 않으며, KISA 공식 복구도구를 복제하지 않는다. 대신 안전한 테스트 파일 환경에서 Rhysida 암호화 구조를 분석하고, 공개 원리 기반 복호화 파이프라인을 교육용/분석용으로 재현하는 것을 목표로 한다.
