# 1. 분석 목적

본 문서는 Rhysida 랜섬웨어 바이너리의 정적 속성과 어셈블리 레벨의 제어 흐름(Control Flow)을 파악하고, 전체 암호화 파이프라인 중 정적 분석 단계에서 도출할 수 있는 핵심 지표 및 메모리 오프셋 정보를 정리하는 것을 목적으로 한다.
Rhysida 정적 분석의 핵심은 동적 실행 없이 바이너리의 기본 구조, 하드코딩된 메타데이터, 감염 제외 로직, IAT(Import Address Table) 내 핵심 API, 그리고 메인 암호화 루프의 진입점(Entry Point)을 식별하는 것이다.

본 프로젝트에서는 Rhysida의 바이너리 구조를 다음 관점에서 분석한다.

1. PE 헤더 및 섹션 엔트로피 분석

2. 하드코딩 문자열 및 예외 처리(Filtering) 분기 로직

3. IAT 기반 주요 Windows API 호출 구조

4. 디렉토리 순회 및 메인 암호화 루프 제어 흐름

5. 간헐적 암호화(Intermittent Encryption) 연산 구조 및 레지스터 상태

6. 암호화 알고리즘 분석 파트와의 데이터 연계

# 2. Rhysida 정적 분석 흐름 개요

본 프로젝트에서 정리하는 Rhysida의 정적 분석 흐름은 다음과 같다.

1. PE 바이너리 기본 헤더 및 엔트로피 확인 (패킹 여부 검증)
2. 섹션 매핑 (.text, .data, .rsrc) 및 메모리 레이아웃 파악
3. 하드코딩된 메타데이터 추출 (랜섬노트, RSA 공개키, C2 흔적)
4. 감염 제외 대상(특정 시스템 폴더 및 파일 확장자) 조건문 추적
5. IAT 매핑 (파일 I/O, 뮤텍스, PRNG 관련 API 식별)
6. 메인 함수 및 디렉토리 순회 루틴 어셈블리 분석
7. 간헐적 암호화 루프 내 블록 오프셋 계산 연산 추적
8. 메인 루프 진입점 및 레지스터 상태 정리 후 팀 공유

이 중 암호화 알고리즘 분석 및 복호화 파이프라인과 직접적으로 연결되는 정적 분석 핵심 요소는 다음 세 가지이다.

-예외 처리 로직 추적 → 암호화 타겟 선별 기준 파악
-IAT & API 매핑 → Key/IV 생성 및 파일 I/O 시점 특정
-메인 루프 어셈블리 분석 → 간헐적 암호화 오프셋 및 레지스터 구조 전달

# 3. PE 헤더 및 섹션 구조

Rhysida 바이너리는 64비트 PE32+ 실행 파일 형태로 제공되며, 정적 분석 도구(Detect It Easy, IDA Pro)를 통한 헤더 분석 결과는 다음과 같다.

|항목||상세 분석 내용|||비고|
|Architecture||x86-64 (64-bit Executable)||Windows 64-bit 환경 타겟|
|Packing / Entropy||패킹 미적용 (낮은 섹션 엔트로피)||코드가 난독화되지 않고 노출됨|
|.text Section||실행 가능 코드 영역||어셈블리 분석 및 제어 흐름 추적 대상|
|.data Section||전역 변수 및 상수 데이터||"문자열, 예외 처리 타겟 리스트 저장"|
|.rsrc Section||리소스 영역||하드코딩된 RSA-4096 공개키 및 랜섬노트 저장|

분석 결과, 상용 패커(Packer)나 복잡한 난독화 기법이 적용되지 않아 IDA Pro 및 Ghidra를 통한 디스어셈블 및 디컴파일이 용이한 구조임을 확인하였다.

# 4. 하드코딩 데이터 및 예외 처리(Filtering) 로직

Rhysida는 시스템의 치명적인 부팅 불능 상태를 방지하고 감염 효율을 높이기 위해 특정 폴더 및 파일 확장자를 암호화 대상에서 제외한다.

[디렉토리 탐색] 
      │
      ▼
[경로 문자열 추출] ──► [하드코딩 예외 리스트와 비교 (cmp/strncmp)]
                              │
                    ┌─────────┴─────────┐
                    ▼                   ▼
            [일치: 암호화 패스]    [불일치: 암호화 수행]


정적 분석을 통해 확인된 하드코딩 예외 처리 대상 및 레지스터 연산 구조는 다음과 같다.          

1. 감염 제외 디렉토리: \Windows, \Program Files, \ProgramData 등 시스템 필수 폴더
2. 감염 제외 확장자: .exe, .dll, .sys, .iso 등 시스템 실행 파일 및 이미지
3. 분기 로직 어셈블리: 문자열 비교 함수 호출 전후로 스택 및 레지스터(RCX, RDX 등)에 경로 인자값이 로드되며, 조건부 점프 명령어(je, jne)를 통해 암호화 루틴 스킵 여부가 결정된다.

4. # 5. IAT 및 주요 API 분석

5. Import Address Table(IAT) 분석을 통해 Rhysida가 파일 시스템 제어 및 난수 생성을 위해 사용하는 핵심 Windows API 목록을 식별하였다.

|분류||API 이름||정적 분석에서의 용도|
|프로세스 제어||CreateMutexW||중복 실행 방지를 위한 시스템 뮤텍스 생성|
|파일 탐색||FindFirstFileW / FindNextFileW||재귀적 디렉토리 및 파일 목록 탐색|
|파일 I/O||"CreateFileW, ReadFile, WriteFile"||"파일 핸들 생성,본문 읽기 및 암호문 덮어쓰기"|
|오프셋 제어||SetFilePointerEx||간헐적 암호화 수행을 위한 파일 포인터 이동|
|난수 생성||CryptGenRandom / SystemFunction036||PRNG 시드 및 내부 상태 연산 관련 API 호출|

이 중 SetFilePointerEx 및 WriteFile API 호출 시점의 전달 방식을 분석함으로써 암호화 루프가 구동되는 방식을 어셈블리 레벨에서 특정할 수 있다.

# 6. 디렉토리 순회 및 메인 루프 분석
Rhysida는 재귀 호출(Recursive Call) 또는 스택 기반 탐색 구조를 사용하여 로컬 드라이브 및 네트워크 공유 폴더를 순회한다.

+-----------------------------------+
|  FindFirstFileW / FindNextFileW   |  ◄── 디렉토리 탐색
+-----------------------------------+
                  │
                  ▼
+-----------------------------------+
|   예외 처리 검증 (폴더 / 확장자)   |
+-----------------------------------+
                  │ (암호화 대상인 경우)
                  ▼
+-----------------------------------+
| CreateFileW (파일 핸들 획득)       |
+-----------------------------------+
                  │
                  ▼
+-----------------------------------+
|  메인 암호화 루프 진입 (Entry)    |  ◄── 핵심 분석 포인트
+-----------------------------------+

정적 분석 관점에서는 메인 암호화 루프 진입 전후의 스택 프레임(Stack Frame)과 파일 핸들(HANDLE)이 전달되는 레지스터(RCX 또는 RDI) 상태를 정확히 매핑하는 것이 중요하다.

# 7. 간헐적 암호화(Intermittent Encryption) 연산 구조

Rhysida는 전체 파일을 암호화하지 않고, 파일 크기(File Size)에 따라 일정 비율의 블록 오프셋을 계산하여 암호화를 건너뛰는 간헐적 암호화 방식을 채택하고 있다.

|구분||연산 방식 및 어셈블리 특징|
|파일 크기 참조||GetFileSizeEx 또는 CreateFileW 후 파일 구조체에서 크기 값 획득|
|오프셋 계산||"파일 크기 대비 암호화 비율(예: 20%, 50% 등) 산술 연산 (imul, idiv, shift)"|
|포인터 이동||계산된 오프셋 값을 SetFilePointerEx API 파라미터로 전달|
|버퍼 처리||암호화할 블록 단위로 ReadFile -> 암호화 -> WriteFile 반복 수행|

이러한 산술 명령어 및 포인터 제어 루틴을 추적하여, 바이너리 레벨에서 어떤 파일 크기 구간에서 어떤 오프셋 공식이 적용되는지 어셈블리 연산식을 규명한다.

# 8. 정적 분석 기반 복호화 연계 조건

정적 분석에서 추출된 바이너리 정보는 암호화 알고리즘 분석 및 복호화 도구 설계 단계에 다음과 같이 활용된다.

1. 메인 루프 진입점 주소 확보: 암호화 수행 직전의 메모리 주소(Entry Point)를 특정하여 동적 검증 및 시뮬레이션 지점 제공
2. API 파라미터 및 레지스터 매핑: PRNG 호출 및 파일 핸들 생성 시점의 레지스터 변화 파악
3. 파일 구조 Footer 매핑: 파일 암호화 완료 후 파일 끝에 연달아 쓰여지는 RSA 암호화된 Key/IV 블록의 크기 및 추가 오프셋 위치 정적 검증
4. 암호화 대상 식별 로직 제공: 복호화 툴 제작 시 어떤 확장자와 구조를 가진 파일이 타겟인지 역으로 필터링하는 기준 설정

# 정리

Rhysida 정적 분석의 핵심은 바이너리의 어셈블리 코드와 PE 구조를 분석하여 암호화 제어 흐름과 예외 처리 로직을 명확히 규명하는 것이다.
본 프로젝트에서 정리하는 정적 분석 구조는 다음과 같다.

-PE 헤더 및 IAT 분석 → 기본 실행 환경 및 API 식별
-예외 처리 & 순회 로직 → 암호화 타겟 선별 흐름 파악
-메인 루프 & 오프셋 연산 → 간헐적 암호화 및 파일 포인터 제어 구조 전달

따라서 정적 분석은 암호화 알고리즘 담당자가 PRNG 시드 역산 및 복호화 파이프라인을 설계할 수 있도록 바이너리 레벨의 기초 데이터와 매핑 주소를 제공하는 핵심적인 역할을 수행한다.
